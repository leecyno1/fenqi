# 分歧 运营 Runbook

## 1. 首次上线
1. 配置生产环境变量。
2. 运行数据库迁移。
3. 用 `pnpm bootstrap:admin` 创建首个管理员。
4. 配置 cron：目录同步 6h、新闻增强 6h、盘口同步 15m、快照 1h。
5. 打开后台检查系统状态卡，确认四类作业都有最新成功记录。

## 2. 常见日常操作
### 创建或编辑事件
- 进入 `/admin`
- 录入题面、锁盘时间、结算来源、图片和状态
- 发布前确认题面、时间边界和来源一致

### 处理结算
- 进入 `/admin/settlements`
- 填写结果、来源名称、来源链接和说明
- 提交后确认详情页和钱包积分已更新

## 3. 同步失败排查
### Polymarket 目录或价格同步失败
1. 查看 `/admin` 系统状态卡里的最近作业记录。
2. 检查 `CRON_SECRET`、`DATABASE_URL` 与外网连通性。
3. 手动触发对应 cron 接口，观察返回错误。
4. 若外部价格源失败，前台应继续显示最近一次有效盘口并标记为 stale。

### 新闻增强失败
1. 检查外部新闻源是否可访问。
2. 确认图片缓存目录或对象存储可写。
3. 若增强失败，事件仍可继续展示，但会保留旧图或 fallback 图。

## 4. 数据 stale 处理
- `sync-polymarket-prices` 超过 30 分钟无成功记录时，`/api/health/ready` 会降为失败。
- 优先恢复价格同步，再确认后台只读状态卡中的更新时间恢复正常。
- 在恢复前，不要把 stale 事件切到纯本地模式，避免展示价格与外部锚点持续偏离。

## 5. 回滚步骤
1. 暂停外部同步 cron。
2. 回滚应用版本。
3. 若 schema 已变化，按迁移版本回滚数据库或恢复备份。
4. 验证 `/api/health/live`、`/api/health/ready`、首页、详情、后台是否恢复。

## 6. 发布前检查单
- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`
- Chrome DevTools 抽查：首页、登录、详情、组合、后台、法律页、健康接口
